## 배경

KBO 직관 기록 다이어리 앱(fullcount.kr)의 초기 이탈률을 낮추고 신규 사용자가 코어 기능을 즉시 인지할 수 있도록 온보딩 튜토리얼을 도입하려고 합니다. 현재 첫 실행 시 응원팀 선택 화면 하나만 있고 기능에 대한 안내는 전혀 없습니다.

---

## 제안: 2파트 온보딩

### 파트 A: 앱 진입 전 슬라이드 (1장)

기존의 단일 "팀 선택" 화면 앞에 1장의 기능 소개 슬라이드를 추가합니다.

- **Step 0**: 앱 소개 — "야구 팬을 위한 완벽한 다이어리 (승률 계산, 50여 개의 밈 배지 컬렉션, 스티커 꾸미기)"
- **Step 1**: 팀 선택 (현재 UI 그대로)

UX:
- 좌우 스와이프 또는 하단 "다음" 버튼으로 진행
- 우측 상단 "건너뛰기(Skip)" 버튼 → 팀 선택으로 즉시 이동
- 기존 유저(index.tsx의 myTeam 체크)는 전체 건너뜀

### 파트 B: 홈 진입 후 코치 마크 (2개 대상)

팀 선택 후 홈에 진입하면, **사용자가 현재 머물고 있는 탭에 따라** 코치 마크가 순차적으로 등장합니다. 무거운 외부 라이브러리 없이 Modal + Absolute Positioning으로 자체 제작합니다.

**홈 탭 (처음 진입 시):**
1. 날짜 스트립 (상단): "좌우로 스와이프하여 날짜를 이동하세요"
2. FAB 버튼 (우측 하단): "여기를 눌러 직관/집관을 기록하세요"

**다이어리 탭 (처음 진입 시):**
3. 다이어리 타임라인 영역: "작성한 일기가 타임라인으로 정리됩니다"

각 탭별로 시퀀셜하게 진행되며, 한 번에 하나의 요소만 집중적으로 가리킵니다.

### 타이밍

[의견 필요] 코치 마크와 **attend_first(👋 첫방문) 배지 토스트**의 순서에 대한 논의가 있었습니다.

- **안 A**: 홈 진입 → 코치 마크 먼저 → 코치 마크 종료 후 토스트
  - 근거: 사용자가 "뭘 할 수 있는지" 먼저 알고, 그 다음에 보상
- **안 B**: 홈 진입 → 토스트 먼저 → 토스트 사라진 후 코치 마크
  - 근거: 환영 메시지 먼저, 자연스러운 딜레이 후 가이드

### 데이터 저장

`user_settings` 테이블에 각 탭별로 본 플래그 저장:
- `has_seen_home_coach` (boolean)
- `has_seen_diary_coach` (boolean)

한 번 보면 다시 안 뜹니다. 부분 완료는 저장하지 않습니다 (코치 마크 도중 이탈 시에도 다시 안 뜸).

## 질문

### Q1. 코치 마크 종료 조건

코치 마크 도중 사용자가 직접 가리켜진 UI(예: FAB)를 누르면 어떻게 해야 할까요?

- **옵션 A**: 사용자가 코치 마크의 X 버튼이나 "다음"을 눌러서만 종료. 도중에 FAB를 눌러도 코치 마크가 끝날 때까지 무시하거나 가이드 종료
- **옵션 B**: FAB를 누르는 즉시 코치 마크 종료 + 해당 액션(일기 쓰기)으로 진입. 더 자연스러운 흐름

### Q2. attend_first 배지 해금 타이밍 (위 타이밍 이슈)

어느 안이 더 좋을까요?

### Q3. 다이어리 탭 코치 마크 필요성

사용자가 첫 일기를 작성한 후 "작성한 게 어디 있지?" 하면서 자연스럽게 다이어리 탭을 발견할 수도 있어서, 굳이 첫 진입 시 가이드할 필요가 없을 수도 있습니다. 다이어리 탭 코치 마크를 유지할지, 아니면 생략할지?

### Q4. 다른 탭 (구장안내, 응원, 순위, MY) 코치 마크

이 탭들은 UI가 직관적이어서 코치 마크가 없어도 된다고 보는데, 혹시 추가가 필요한 탭이 있을까요?
